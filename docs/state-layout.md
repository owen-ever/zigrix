# State Layout

## Directory shape
```text
~/.zigrix/zigrix.config.json        # config location
<paths.baseDir>/
├─ tasks/
│  ├─ DEV-YYYYMMDD-NNN.meta.json   # machine-readable metadata
│  └─ DEV-YYYYMMDD-NNN.md          # human-readable spec
├─ prompts/
├─ evidence/
│  └─ DEV-YYYYMMDD-NNN/
│     ├─ <agent>.json
│     └─ _merged.json
├─ rules/
│  ├─ orchestrator-agent.md
│  ├─ frontend-agent.md
│  └─ ...
├─ runs/
│  └─ run-*.json
├─ tasks.jsonl
└─ index.json
```

## Source-of-truth files
- `tasks/*.meta.json`: task metadata (machine-readable)
- `tasks/*.md`: task spec (human-readable, editable)
- `tasks.jsonl`: append-only event ledger
- `evidence/<taskId>/*.json`: per-agent evidence
- `rules/*.md`: agent rule files (seeded from orchestration/rules/)
- `zigrix.config.json`: full config

## Derived files
- `index.json` — rebuildable via `zigrix index-rebuild`
- `_merged.json` — generated by `zigrix evidence merge`
- rendered prompts under `prompts/`
- workflow run records under `runs/`

## Legacy compatibility
- `tasks/*.json` (old single-file format) is read with fallback
- Legacy events with `timestamp` field are normalized to `ts`

## Operator guidance
- If `index.json` looks wrong, rebuild it with `zigrix index-rebuild`.
- If one template/rule is broken, reset that subtree instead of nuking the whole config.
- If runtime state is irrecoverably messy, `zigrix reset state --yes` is the clean fallback.
